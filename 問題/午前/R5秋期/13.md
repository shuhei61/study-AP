<p class="ap-question"><strong>問13</strong>　システムの性能を向上させるための方法として，スケールアウトが適しているシステムはどれか。</p>
<ul class="ap-choices">
<li class="ap-choice-item ap-wrong">
<div class="ap-choice-text">ア　一連の大きな処理を一括して実行しなければならないので，並列処理が困難な処理が中心のシステム</div>
<div class="ap-choice-note">並列処理が困難なので、<a href="用語/スケールアウト" class="internal-link" data-href="用語/スケールアウト">スケールアウト</a>は適しません。</div>
</li>
<li class="ap-choice-item ap-correct">
<div class="ap-choice-text">イ　参照系のトランザクションが多いので，複数のサーバで分散処理を行っているシステム</div>
<div class="ap-choice-note">正しい。参照系がメインであり、現状でも並列処理を行っているので、<a href="用語/スケールアウト" class="internal-link" data-href="用語/スケールアウト">スケールアウト</a>での<a href="用語/性能" class="internal-link" data-href="用語/性能">性能</a>向上が適しています。</div>
</li>
<li class="ap-choice-item ap-wrong">
<div class="ap-choice-text">ウ　データを追加するトランザクションが多いので，データの整合性を取るためのオーバーヘッドを小さくしなければならないシステム</div>
<div class="ap-choice-note">整合性を維持するための処理を少なくしたいので、<a href="用語/スケールアウト" class="internal-link" data-href="用語/スケールアウト">スケールアウト</a>は適しません。</div>
</li>
<li class="ap-choice-item ap-wrong">
<div class="ap-choice-text">エ　同一のマスターデータベースがシステム内に複数配置されているので，マスターを更新する際にはデータベース間で整合性を保持しなければならないシステム</div>
<div class="ap-choice-note">整合性の保持をしなければならないので、<a href="用語/スケールアウト" class="internal-link" data-href="用語/スケールアウト">スケールアウト</a>は適しません。</div>
</li>
</ul>
<div class="ap-explanation">
<h4>解説</h4>
<p>システムの<a href="用語/処理能力" class="internal-link" data-href="用語/処理能力">処理能力</a>を向上させる施策には、「<a href="用語/スケールアップ" class="internal-link" data-href="用語/スケールアップ">スケールアップ</a>」と「<a href="用語/スケールアウト" class="internal-link" data-href="用語/スケールアウト">スケールアウト</a>」という2つのアプローチがあります。</p>
<p><a href="用語/スケールアップ" class="internal-link" data-href="用語/スケールアップ">スケールアップ</a>は、<a href="用語/サーバ" class="internal-link" data-href="用語/サーバ">サーバ</a>を構成する各処理装置をより<a href="用語/性能" class="internal-link" data-href="用語/性能">性能</a>の高いものに交換したり、プロセッサの数などを増やすことで<a href="用語/サーバ" class="internal-link" data-href="用語/サーバ">サーバ</a>当たりの<a href="用語/処理能力" class="internal-link" data-href="用語/処理能力">処理能力</a>を向上させる方法です。追加・削除・更新処理が頻繁に発生するなど、並列実行による負荷<a href="用語/分散" class="internal-link" data-href="用語/分散">分散</a>が困難なシステムに適しています。</p>
<p><a href="用語/スケールアウト" class="internal-link" data-href="用語/スケールアウト">スケールアウト</a>は、接続する<a href="用語/サーバ" class="internal-link" data-href="用語/サーバ">サーバ</a>の台数を増やすことで<a href="用語/サーバ" class="internal-link" data-href="用語/サーバ">サーバ</a>システム全体としての<a href="用語/処理能力" class="internal-link" data-href="用語/処理能力">処理能力</a>や<a href="用語/可用性" class="internal-link" data-href="用語/可用性">可用性</a>を向上させる方法です。同等の<a href="用語/性能" class="internal-link" data-href="用語/性能">性能</a>アップであれば<a href="用語/スケールアップ" class="internal-link" data-href="用語/スケールアップ">スケールアップ</a>よりも低コストであることが多いです。更新処理（追加や削除も含む）を伴う機能を複数の<a href="用語/サーバ" class="internal-link" data-href="用語/サーバ">サーバ</a>に負荷<a href="用語/分散" class="internal-link" data-href="用語/分散">分散</a>すると、データの整合性を維持するための処理や<a href="用語/排他制御" class="internal-link" data-href="用語/排他制御">排他制御</a>による遅延が生じてしまい、効果的な<a href="用語/性能" class="internal-link" data-href="用語/性能">性能</a>向上につながりません。このため、更新処理が主要なシステムに対しては<a href="用語/スケールアウト" class="internal-link" data-href="用語/スケールアウト">スケールアウト</a>は不向きです。</p>
<p><a href="用語/スケールアウト" class="internal-link" data-href="用語/スケールアウト">スケールアウト</a>が適しているのは、更新処理の整合性要件が緩く、個々に独立して並列実行できる処理を扱うシステムです。</p>
