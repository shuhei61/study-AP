# 過去問ノート作成 — オーケストレーター（親エージェント用）

あなたは**過去問ノートを自分では書かず**、サブエージェントに1問ずつ作成を任せ、完了後の照合と集計だけを行う**オーケストレーター**です。

## 前提

- 手順の正本: `docs/ai/過去問ノート.md`（ワーカーにも同じルールを渡す）
- ワーカー用プロンプト雛形: `docs/prompts/過去問ノート-worker.md`
- サブエージェント種別: **`generalPurpose`**（`explore` は読み取り専用のため不可）

## 作業開始前（親）

計画・起動・検証の**前に**、ボルト内のファイルを開いて読む（`README.md` 全文をワーカー prompt に貼る必要はない）。

1. `README.md` を読む
2. 本ファイル（`docs/prompts/過去問ノート-orchestrator.md`）
3. 検証・集計のため `docs/ai/過去問ノート.md` の手順 2.5・6〜7（図はワーカーが `fetch_question_figures.py`。**`count_term_question_links.py` は親が実行**）

読み終えてから、以下の手順に進む。

## ユーザー依頼から決めること

依頼文から次を確定し、不足があればユーザーに聞く（推測で埋めない）。

| 項目 | 例 |
|------|-----|
| 試験区分・回 | `午前` / `R3春期` または `R3秋期` など |
| 問番号 | `7` または `7〜10` |
| ap-siken URL 規則 | `https://www.ap-siken.com/kakomon/{スラッグ}/q{N}.html`（例: 春 `03_haru`、秋 `03_aki`） |
| 保存パス | `問題/午前/{試験回}/{N}.md`（ファイル名に `問` は付けない） |

試験回とスラッグの対応表: [`docs/ai/試験回とパス.md`](../ai/試験回とパス.md)

## オーケストレーターの手順

### 1. 計画

- 対象の各問について、**既に `問題/…/{N}.md` があるか**確認する。ある場合はスキップするかユーザーに確認する。
- 複数問は **必ずサブエージェント**で進める（**1 Task = 1問**。親が複数問を連続で自分で書く **親一括** は、下記「親一括」のとおりユーザー明示時のみ）。
- **Task を並列起動:** 問ごとに **別々の `Task` を1回ずつ**起動する（**1 Task に複数問を書かない** — Cursor 停止の主因）。
- 同時起動: **既定は最大4 Task**（= 最大4問ぶんの **独立した Task 4本**。例: 問57・58・59・60 なら **Task×4**。「問57〜60を1 Task で」は禁止）。停止が続くときだけ **最大2 Task** に下げる。
- **HTTP:** 各問で `--print-source` は **1回だけ**。`--apply` は一次 HTML のみ（再 HTTP しない）。**WebFetch / ブラウザ / `curl` 禁止**（下記「ワーカー起動時の追記」）。

### 2. サブエージェント起動

各問ごとに `Task`（`subagent_type: generalPurpose`）を起動する。

- **description**: 短いタイトル（例: `R3春 問7 過去問`）
- **prompt**: `docs/prompts/過去問ノート-worker.md` の内容をコピーし、プレースホルダを**すべて**置換して渡す（**省略しない**。特に手順1・4のスクリプト1行実行を必ず含める）
- **prompt 末尾に必ず追記**（短縮プロンプトでも省略しない）:
  ```
  【厳守】ap-siken は python3 scripts/fetch_question_figures.py のみ（--print-source 1回 → --apply）。
  WebFetch・ブラウザ・curl・wget・スクリプト import は使わない。URL を開く許可要求は出さない（出ても拒否してスクリプトに戻る）。
  ```
- ワーカーに `README.md` 全文は渡さない（ワーカーは `過去問ノート-worker.md` の必読どおり、ボルト内の `README.md` を自分で読む）

### 3. ワーカーに任せないこと（親が最後にまとめて実行）

並列実行時のファイル競合を避けるため、ワーカーは次を**実行しない**。親が全ワーカー完了後に **1回だけ** 実行する。

- `count_term_question_links.py`
- `clear_ap_siken_html_cache.py`（`workspace/ap-siken-html/` の一次 HTML 削除）

ワーカーが実行しないもの（親も普段実行しない）:

- `build_tag_index.py`
- `build_glossary_index.py`
- `用語/○○.md` の新規作成

### 4. 完了後の検証（親）

各問のパスに対し（**`cd`・`&&` 連結なし**。コマンドは1行ずつ）:

```bash
python3 scripts/check_question_terms.py --question 問題/午前/R3春期/7.md
```

（図の取得はワーカー手順 2.5 で済んでいる想定。未実施の問があれば `python3 scripts/fetch_question_figures.py --apply --question 問題/…` を親が実行してよい）

- 終了コード 1 や一覧外リンクがあれば、**親が最小修正**するか、該当問だけワーカーを再実行する。
- **文中インライン図**（番号付き条件＋条件文中の小図）の問は、`docs/ai/過去問ノート.md` **手順 4.5** を満たしているか目視する。未対応なら「問16パターンで直して」とワーカー再実行または親が最小修正。
- 全問 OK になったら **1回だけ**（この順）:

```bash
python3 scripts/count_term_question_links.py
python3 scripts/clear_ap_siken_html_cache.py
```

### 5. ユーザーへの報告

次の形式で簡潔にまとめる（長い解説は不要）。

- 作成したファイル一覧（パス + ap-siken URL）
- 正解（ア〜エ）
- スキップ・再実行した問があれば理由
- 照合結果（OK / 修正した点）
- `用語一覧.md` の（N）更新と一次 HTML キャッシュ削除を実行した旨

## 親一括（親が複数問を自分で書く）

**既定では行わない。** 件数が多くても、品質と手順の隔離のため **1 Task = 1問・同時最大4 Task** のサブエージェント運用を続ける。

親が `fetch --print-source` から用語リンク・照合までを、複数問ぶん自分で繰り返す **親一括** は、ユーザーが次を**明示したときだけ**許可する。

- 「親が一括で」「サブエージェントを使わず」「オーケストレーションせず直接作成」など
- 停止回避のため親一括を指示したとき

明示がないのに親一括に切り替えてはいけない例:

- 問数が多い（例: 問37〜80）からといって親がまとめて書く
- Cursor の停止が続くからといって、ユーザーに確認せず親一括にする
- サブエージェントが1回失敗した直後に、該当問の再 Task を試さず親一括にする

停止や失敗時は、まず **同時 Task 数を下げる（4→2）** または **1回に任せる問数を減らす（例: 10問ずつ）**・**該当問だけ Task を再実行**を検討する。ワーカーが WebFetch を使った報告があれば **その問だけ再 Task**（prompt 末尾の【厳守】を確認）。親一括が必要ならユーザーに確認する。

## 1問だけ親が直接書いてよい場合

**親一括ではない**（1ファイルのみ）。次のときはサブエージェントを使わず、親が `docs/ai/過去問ノート.md` に従って直接作成してよい。

- **1問だけ**の依頼で、ユーザーがオーケストレーター・並列・サブエージェントを指定していない

## 禁止・注意

- **親一括**（複数問を親が連続作成）— ユーザー明示がない限り禁止
- 親がワーカーと同じファイルを同時に編集しない
- **1 Task に複数問**（「問37〜40を順に」「この Task で問57〜60」等）— **4問並列 ≠ 1 Task に4問**。長時間化し Cursor が停止しやすい
- ap-siken は **`fetch_question_figures.py` のみ**（`--print-source` で本文、`--apply` で図）。WebFetch / ブラウザ / `curl` / スクリプト import 禁止（ワーカー prompt 末尾の【厳守】を省略しない）
- スクリプト実行は **`python3 scripts/…` の1行のみ**（`cd`・`&&` 連結を付けさせない）
- HTML ブロック内に空行を入れない（ワーカー成果物を直すときも同様）
- ユーザーの明示がない限り git commit しない
